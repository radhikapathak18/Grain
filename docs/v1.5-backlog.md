# Grain v1.5 backlog

Items that were explicitly cut from v1 to keep the 24-hour build inside the
demo flow (see `Docs/grain_24hr_hackathon_plan.md` "what's in scope" and the
"no" list in `Docs/phase_0_decisions.md` §7). Each is captured here with
**why it was deferred** and **what real-usage signal would qualify it for
v1.5 vs. waiting for v2**.

The promotion test is the same for every item: *during v1 pilot usage, do
we see a behavior that the missing capability would have prevented or
unlocked?* If yes → v1.5. If the signal is theoretical or only emerges
from internal speculation → v2.

---

### 1. Cross-product taxonomy bridging via embedding similarity

Today the only cross-product join is exact match on `area` (e.g. an
"onboarding" claim in Helix Core links to an "onboarding" claim in P4V).
Embedding similarity would surface latent links — "view-spec setup" in
Helix Core ↔ "workspace wizard" in P4V even though the area strings
differ.

**Why deferred:** the v1 corpus is small enough that a hand-curated
flat-string taxonomy carries the demo; adding an embedding index touches
ingest, storage, and retrieval all at once.

**Promote to v1.5 if:** PMs in pilot usage repeatedly ask follow-up
questions that imply they expected two area buckets to be the same
("you said wizard friction for P4V, what does that look like in Helix
Core?") and Grain has to be re-prompted to connect them.

### 2. Live Pendo querying at synthesis time

Right now Pendo "evidence" is pre-extracted aggregations baked into the
fixture corpus. The vision is to fan out a Pendo query when synthesis
needs a quantitative datum that isn't already in the claim store.

**Why deferred:** every live-query path adds a real auth surface, a
real latency budget, and a real failure mode mid-stream — none of which
we could carry inside 24 hours.

**Promote to v1.5 if:** in pilot usage we see researchers asking
"what's the abandonment rate THIS week?" type questions where the
already-extracted Pendo claim is stale enough to mislead. If pilot
users only ever cite the existing aggregation, hold for v2.

### 3. Two-layer entity model (canonical + extracted)

Claims today are flat — `customer: "Stellar Forge Games"` is a string
on the evidence record. A canonical layer would let "Stellar Forge",
"Stellar Forge Games", and "SFG" resolve to one entity for filtering
and counts.

**Why deferred:** v1 controls the fixture spelling end-to-end, so the
de-duplication problem is artificial. Phase 0 §7 explicitly cuts it.

**Promote to v1.5 if:** once real connectors land (item 7), we see the
same customer or persona spelled multiple ways in claim attribution
within the first month. If real ingest produces clean canonical names
on its own, hold for v2.

### 4. Human review queue

Low-confidence extractions today are not gated — they go straight into
`CLAIMS` and surface in synthesis with whatever trust tier their source
type implies. The roadmap calls for a queue where humans confirm or
reject borderline claims before they ship.

**Why deferred:** there is no extraction pipeline in v1, so there is
nothing to queue. Reviewing pre-curated demo claims would be theater.

**Promote to v1.5 if:** real extraction (item 7) lands and a pilot user
flags a claim as "this isn't what the customer said." That feedback is
the v1.5 product — close the loop with a queue. Until then, hold.

### 5. Computed confidence scoring

Trust tier today is derived from the **highest** source tier on a
claim's evidence array (T1 if any zoom, T2 if any gong/pendo, T3
otherwise). A computed score would factor evidence count, recency,
cross-source agreement, and contradiction.

**Why deferred:** the rule-based tier already produces visually
credible badges on the fixture corpus; building a real score requires
real evidence-volume distributions we don't have yet.

**Promote to v1.5 if:** pilot users look at a tier-1 claim with only
one piece of evidence and treat it as if it were a tier-1 claim with
five — that's the moment a computed score would have done useful work.

### 6. Full eight-axis tagging

The architecture plan called for eight axes (product, area, persona,
sentiment, stakeholder type, life-cycle stage, etc.). v1 ships with
four — `product`, `area`, `persona`, `sentiment` — because those four
are all the demo flow exercises.

**Why deferred:** the four extra axes only matter for filtered queries
("show me only post-purchase complaints from build engineers"), which
the v1 chat UI doesn't support.

**Promote to v1.5 if:** pilot users ask shape-of-question that the
existing four axes can't slice (most likely: stakeholder type when
mixing internal sales-call notes with external research). If they
don't, hold the additional axes until claims browser ships.

### 7. Real MCP connectors

v1 reads from `apps/api/src/data/sources/*.ts` — hand-curated
fixtures. v1.5+ would ingest from real Gong, Slack, Confluence, Pendo,
and Zoom via MCP.

**Why deferred:** every connector is its own auth + rate-limit + schema
project. Phase 0 §7 cuts all real connectors explicitly.

**Promote to v1.5 if:** after the demo, the first stakeholder ask is
"can I point this at our actual Gong account?" — that's the moment to
prioritize one (likely Gong) and ship a thin pipeline. Going to v2
without at least one real connector is hard to justify.

### 8. Hybrid context inheritance

Right now a session's product context is set once at login (multi-
select) and held flat for the session. Hybrid inheritance would let
context propagate down from a parent question ("about Helix Core") to
narrowed follow-ups ("specifically about onboarding") while still
allowing the user to broaden again mid-thread.

**Why deferred:** the demo flow only requires one product-context
shape (the multi-select set at login). Mid-session context changes
are explicitly excluded in `grain_24hr_hackathon_plan.md` "what's in
scope."

**Promote to v1.5 if:** pilot users repeatedly send "actually, restrict
that to just P4V" follow-ups and the synthesis ignores them — that's
the inheritance gap showing through. If users restart the session to
change context, hold.

### 9. Multi-product context narrowing/broadening mid-session

Closely related to item 8 but specifically about the *chip* in the
header. Today it's read-only mid-chat; v1.5 would let a user click it
to drop a product from the active query without losing chat history.

**Why deferred:** the chip surface exists but is informational only.
Adding mutation requires deciding what happens to prior turns in the
thread (do they get re-asked? grayed out?) — a UX decision that
deserves its own design pass, not a corner-cut at hour 22.

**Promote to v1.5 if:** in pilot use, researchers consistently start a
session over to drop a product, rather than asking the same question
inside the existing context. If the friction is real, the chip needs
to be writeable. If users live with the multi-product blend, hold.

---

**Items NOT on this list intentionally:** anything in the Phase 0 "no"
list that we still believe is correct to omit at v2 either (no
PostgreSQL until claim volume crosses 500k per Phase 0 §7; no real
OAuth until Grain leaves internal pilot). Those are not v1.5 questions —
they are platform questions that wait for a different signal.
